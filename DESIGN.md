# Design notes

## Why standards-only

A NIST PQC standard has **fixed** key/signature sizes and **fixed** bytes. You
cannot "optimize" ML-DSA-87 to produce smaller or different signatures and still
be ML-DSA-87 — the result is a non-standard variant that no other implementation
can verify. That non-interoperability is precisely the failure this library
exists to prevent, so `qorechain-pqc` wraps audited implementations of the final
standards and proves they agree byte-for-byte.

Tested cross-implementation results (each binding verifying signatures/secrets
produced by `@noble/post-quantum`, the FIPS reference):

| Primitive | `@noble` (JS) | Rust `fips204/203` | Go CIRCL | C liboqs | Python liboqs | Java BC |
|---|---|---|---|---|---|---|
| ML-DSA 44/65/87 | ✅ | ✅ | ✅ | ✅ | ✅ | builds in CI |
| ML-KEM 512/768/1024 | ✅ | ✅ | ✅ | ✅ | ✅ | builds in CI |
| SHAKE-256 | ✅ | ✅ | ✅ | ✅ | ✅ | builds in CI |

## Optimizing on-chain footprint (without breaking the standard)

The real levers, in order of impact:

1. **Pick the security level deliberately.** ML-DSA-65 (L3) signatures are ~28%
   smaller than ML-DSA-87 (L5) and remain very strong; ML-KEM-768 ciphertexts are
   smaller than 1024. Choose per use-case.
2. **Pay-to-pubkey-hash.** Store only `pubkeyHash(pk)` (20–32 bytes of SHAKE-256)
   in account state and require the full public key in the transaction. Account
   state stays tiny regardless of the 1–2.5 KB key.
3. **Verify-and-discard signatures.** A signature must live in the transaction
   (block data) but should never be written into the persistent state tree.
4. **Implementation speed.** Each binding uses a fast, audited backend; on-chain
   the cost is *verification*, which is the cheap direction for ML-DSA.

> If signature size is the dominant constraint, FN-DSA (Falcon) gives ~1.3 KB L5
> signatures (3.5× smaller than ML-DSA-87). It is intentionally **not** included
> here: FN-DSA is FIPS-206 *draft* (not final) and its signing path needs
> constant-time floating-point care. Revisit once FIPS-206 is finalized.

## Note for the QoreChain node

A live cross-implementation test found the chain's signer
(`pqcrypto-dilithium 0.5`) is a **non-standard** Dilithium variant: it has
ML-DSA-87 *sizes* (pk 2592 / sig 4627) but its signatures do not verify under any
standard FIPS-204 library, in any context mode. Migrating the node's signature
crate to the maintained FIPS-204 successor (`pqcrypto-mldsa`) makes the whole
suite standard and lets this library — and any standard wallet/tooling —
interoperate with the chain directly. The KEM (`pqcrypto-mlkem`, FIPS-203) and
the hash (SHAKE-256, FIPS-202) are already standard and already interoperate.

## Vectors

`vectors/*.json` (per algorithm) and `vectors/flat.txt` (line-based, for the C
and Java tests) are generated by `scripts/gen-vectors.mjs` from
`@noble/post-quantum`. ML-DSA/ML-KEM keygen is deterministic from the recorded
seed (so `keygen(seed)` is also checked where the backend supports it); signature
and decapsulation vectors are verified by every binding.
